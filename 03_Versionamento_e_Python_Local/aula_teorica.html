3

Configure sua identidade no Git

O Git precisa saber quem é você para assinar os commits. Abra o terminal do VS Code (menu Terminal → New Terminal, ou pressione Ctrl + `) e rode os dois comandos abaixo, substituindo pelo seu nome e e-mail:

git config --global user.name "Maria Silva"
git config --global user.email "maria.silva@email.com"

Troque "Maria Silva" pelo seu nome completo e "maria.silva@email.com" pelo seu e-mail real.

📌 Qual e-mail usar? Use o mesmo e-mail que você vai cadastrar na sua conta do GitHub (explicado na Seção 2). Se ainda não tem conta no GitHub, crie a conta primeiro e volte aqui com o e-mail cadastrado.

Após rodar os dois comandos, o terminal vai mostrar só o cursor piscando — isso é normal, significa que funcionou. Para confirmar, rode:

git config --global user.name
git config --global user.email

O terminal vai exibir o nome e e-mail que você digitou. Se estiverem corretos, está tudo certo.

Opção 2 💻 Criar pelo terminal — para quem já tem uma pasta no computador

Você já tem uma pasta com arquivos no seu computador e quer começar a versionar. Aqui você inicializa o Git localmente e conecta ao GitHub depois.

1

Abra a pasta do projeto no VS Code e inicialize o Git

No terminal do VS Code (Terminal → New Terminal), navegue até a pasta do seu projeto e rode:

git init

Isso cria um repositório Git vazio dentro da pasta. Você vai ver uma pasta oculta .git aparecer — é onde o Git guarda todo o histórico.

2

Faça o primeiro commit com os arquivos existentes

git add .
git commit -m "Primeiro commit — início do projeto"

git add . adiciona todos os arquivos da pasta de uma vez. git commit salva o estado atual no histórico.

3

Crie um repositório vazio no GitHub

Acesse github.com/new e crie o repositório — mas desta vez NÃO marque "Add a README file". O repositório precisa estar vazio para receber o seu histórico local.

⚠️ Atenção: se marcar README, o GitHub cria um commit automático e vai dar conflito na hora do push. Repositório vazio = sem nenhuma opção marcada.
4

Conecte o repositório local ao GitHub e envie

Copie a URL do repositório recém-criado no GitHub e rode no terminal:

git remote add origin https://github.com/seu-usuario/nome-do-repo.git
git push -u origin main

O -u origin main é só na primeira vez — configura o vínculo entre o seu computador e o GitHub. Nos próximos pushes basta git push.

Quando o Git pede ajuda: conflito de merge

Quando acontece na prática: você e um colega editam o mesmo arquivo, na mesma linha, e os dois fazem push. O Git não sabe qual versão manter — então ele para e pede que você decida. Na maioria das vezes o Git junta as mudanças automaticamente. Mas quando duas versões alteram a mesma linha de jeitos diferentes, ele marca o trecho no arquivo assim:

<<<<<<< HEAD
total = preco * quantidade
=======
total = preco * quantidade * 1.1  # com imposto
>>>>>>> feature/calculo-imposto
Conflito não é erro — é o Git pedindo sua decisão. No VS Code com GitLens instalado, aparecem botões "Accept Current Change" / "Accept Incoming Change" — clique no que quiser manter. Se preferir resolver manualmente: apague as linhas com <<< / === / >>>, deixe só o trecho que deve ficar e faça um novo commit.

O que o Git não deve guardar: .gitignore

Nem tudo numa pasta de projeto deve ir para o GitHub — senhas, arquivos temporários do sistema, ambientes virtuais. O arquivo .gitignore lista o que o Git deve ignorar completamente.

Como criar o arquivo: no VS Code, clique em File → New File… (Ctrl+N), salve com Ctrl+S e nomeie exatamente como .gitignore (com ponto no início, sem extensão). O VS Code vai perguntar o tipo — escolha Plain Text. Depois cole o conteúdo abaixo:

# .gitignore — copie este modelo para o seu repositório
*.pyc
__pycache__/
.env
venv/
.DS_Store
💡 Nunca coloque senhas ou tokens de API em arquivos versionados. Mesmo que você apague depois, o histórico do Git guarda tudo. Use variáveis de ambiente (.env no .gitignore) para dados sensíveis.

3. Listas, Tuplas e Dicionários

Voltando ao Python — até a Semana 02, cada variável guardava um único valor. Agora você aprende a guardar vários valores organizados numa única estrutura.

EstruturaCaracterísticaExemplo
ListaMutável, ordenada, acessada por posiçãocompras = ["arroz", "feijão"]
TuplaImutável — não pode mudar depois de criadacoordenadas = (10, 20)
DicionárioAcessado por chave (nome), não por posiçãoproduto = {"nome": "Mouse", "preco": 45}

Lista

Exemplo 1 — Criando e acessando uma lista

compras = ["arroz", "feijão", "leite"]

print(compras[0])      # arroz  — posição 0 é sempre o primeiro
print(compras[-1])     # leite  — posição -1 é sempre o último

compras.append("café") # adiciona ao final da lista
print(compras)         # ["arroz", "feijão", "leite", "café"]
⚠️ Erro real do Python — IndexError: a lista compras tem 4 itens (índices 0 a 3). Se você pedir compras[10]: 
IndexError: list index out of range
"Índice fora do intervalo" — você pediu uma posição que não existe. Tente provocar esse erro de propósito no notebook.

Exemplo 2 — Medindo e percorrendo uma lista

notas = [8.5, 7.0, 9.5, 6.0]

print(f"Total de notas: {len(notas)}")   # 4
print(f"Maior nota: {max(notas)}")        # 9.5
print(f"Menor nota: {min(notas)}")        # 6.0

for nota in notas:
    print(nota)                           # imprime cada nota

Dicionário

Exemplo 1 — Acessando valores por chave

produto = {"nome": "Mouse", "preco": 45.0, "estoque": 12}

print(f"{produto['nome']} custa R$ {produto['preco']}")
print(f"Estoque disponível: {produto['estoque']} unidades")

Exemplo 2 — Atualizando um valor no dicionário

aluno = {"nome": "Carlos", "nota": 7.5}

print(f"Nota atual: {aluno['nota']}")   # 7.5

aluno["nota"] = 9.0                     # dicionário é mutável — pode alterar
print(f"Nota corrigida: {aluno['nota']}")  # 9.0
Dicionário ≠ Tupla: ao contrário da tupla, o dicionário permite alterar valores depois de criado. Só o que não muda é a chave.
